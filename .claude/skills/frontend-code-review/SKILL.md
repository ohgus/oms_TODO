# Frontend Code Review

프론트엔드 코드의 설계 품질을 **가독성(Readability)**, **예측 가능성(Predictability)**, **응집도(Cohesion)**, **결합도(Coupling)** 4가지 축으로 평가하고 개선안을 제시하는 스킬.

## 리뷰 워크플로우

1. 코드를 읽고 4축 각각에서 문제가 있는지 식별
2. 문제별로 해당 축 이름, 위반 규칙, 이유를 명시
3. 권장 패턴 코드와 함께 개선안 제시
4. 변경 전/후 비교로 이점 설명

## 4축 요약 & 핵심 체크리스트

### 1. 가독성 (Readability)

| 규칙                      | 체크 포인트                                                       |
| ------------------------- | ----------------------------------------------------------------- |
| 매직 넘버에 이름 붙이기   | 설명 없는 숫자 리터럴이 있는가?                                   |
| 구현 세부사항 추상화      | 인증·다이얼로그 등 복잡한 로직이 컴포넌트에 직접 노출되어 있는가? |
| 조건부 렌더링 분리        | 하나의 컴포넌트가 역할별로 크게 다른 UI를 조건문으로 분기하는가?  |
| 삼항 연산자 단순화        | 중첩 삼항 연산자가 있는가? → IIFE + if/else 권장                  |
| 시선 이동 줄이기          | 단순 로직이 먼 곳에 분리되어 있는가? → 인라인/코로케이션 권장     |
| 복잡한 조건에 이름 붙이기 | 긴 불리언 표현식이 의미 없이 나열되어 있는가?                     |

### 2. 예측 가능성 (Predictability)

| 규칙                       | 체크 포인트                                                 |
| -------------------------- | ----------------------------------------------------------- |
| 반환 타입 표준화           | 유사 함수/훅의 반환 타입이 제각각인가?                      |
| 숨겨진 로직 드러내기 (SRP) | 함수가 이름과 무관한 부작용(로깅 등)을 내부에서 수행하는가? |
| 고유하고 설명적인 이름     | 래퍼 함수명이 원본과 구분 안 되거나 모호한가?               |

### 3. 응집도 (Cohesion)

| 규칙                    | 체크 포인트                                                    |
| ----------------------- | -------------------------------------------------------------- |
| 폼 응집도               | 필드 간 독립/의존 관계에 맞는 유효성 검증 전략을 사용하는가?   |
| 기능/도메인별 코드 조직 | 디렉토리가 코드 유형별(components/, hooks/)로만 나뉘어 있는가? |
| 매직 넘버-로직 연관성   | 상수가 관련 로직과 멀리 떨어져 정의되어 있는가?                |

### 4. 결합도 (Coupling)

| 규칙                | 체크 포인트                                                 |
| ------------------- | ----------------------------------------------------------- |
| 성급한 추상화 지양  | 분기 가능성이 있는 로직을 억지로 하나의 추상화로 묶었는가?  |
| 상태 관리 범위 한정 | 넓은 범위의 훅이 불필요한 리렌더링을 유발하는가?            |
| Props 드릴링 제거   | 중간 컴포넌트가 자신이 사용하지 않는 props를 전달만 하는가? |

## 리뷰 출력 포맷

각 발견 사항을 다음 형식으로 보고:

```
### [축 이름] 위반: [규칙명]

**위치:** `파일명:라인` 또는 해당 코드 블록
**문제:** 무엇이 문제인지 간결히 설명
**개선안:**
- 변경 전 코드 (간략히)
- 변경 후 코드 (권장 패턴 적용)
**이점:** 변경으로 얻는 구체적 이점
```
